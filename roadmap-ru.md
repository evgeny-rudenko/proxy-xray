# План доработок Proxy Xray

Это примерный план будущих доработок проекта. Текущая концепция: домашний LAN-шлюз на Xray с подпиской VLESS, автоматическим выбором рабочих серверов, прямой маршрутизацией российских ресурсов, split DNS, веб-статусом и запуском через Docker Compose.

## 1. Читать результаты Xray Observatory

Сейчас Xray сам проверяет outbounds через `observatory`, но supervisor эти результаты не читает.

Цель:

- показывать реальное состояние outbounds по мнению Xray;
- видеть, какие серверы Xray считает живыми или деградировавшими;
- меньше полагаться на наши косвенные проверки.

Возможная реализация:

- включить Xray API/stats;
- прочитать результаты observatory;
- добавить данные в `/json`;
- отобразить их на веб-странице.

Плюсы:

- более точный список живых серверов;
- меньше собственной логики проверки;
- проще разбирать, почему balancer выбрал конкретный outbound.

Риски:

- нужно аккуратно настроить Xray API;
- формат данных observatory нужно проверить на реальном контейнере.

## 2. Deploy одной командой по SSH

Добавить локальный deploy-скрипт для домашнего сервера:

```sh
./deploy.sh home
```

Ожидаемое поведение:

- читать настройки сервера из `.deploy.env`;
- копировать проект на сервер через `rsync`;
- не затирать `state.json` и `vless-extra.txt`;
- запускать `docker compose config --quiet`;
- запускать `docker compose up -d --build`;
- прогонять smoke-тест;
- печатать итоговый URL статуса и порты прокси.

Такой вариант лучше GitHub Actions, потому что домашний сервер не нужно открывать в интернет.

## 3. Backup и restore операторских файлов

Перед deploy или крупными изменениями сохранять важные файлы:

```text
backups/
  2026-06-06_15-30/
    state.json
    vless-extra.txt
    docker-compose.yml
```

Полезные команды:

- создать backup;
- показать список backup;
- восстановить backup;
- сравнить текущие файлы с backup.

Главная цель: не потерять локальные VLESS-серверы и накопленное состояние проверок.

## 4. Кнопки на странице статуса

Так как страница доступна только в локальной сети, можно добавить несколько простых действий.

Кандидаты:

- обновить подписку сейчас;
- запустить одну случайную проверку сервера сейчас;
- перезапустить Xray;
- очистить failed marks;
- скачать/export `state.json`;
- возможно скачать текущий сгенерированный Xray config.

Важно не превращать страницу в большой admin panel. Она должна оставаться простой.

## 5. Структурированные логи и фильтры

Сейчас логи плоские.

Можно добавить категории:

- health;
- throughput;
- subscription;
- candidate checks;
- Telegram;
- DNS;
- Xray restart/config.

На веб-странице можно показывать последние важные события и фильтровать их. `/logs` можно оставить обычным plain text endpoint.

## 6. Diagnostics endpoint

Добавить отдельный endpoint:

```text
/diagnostics
```

Он должен проверять или показывать:

- `.ru` DNS через RU DNS;
- global DNS через global DNS;
- наличие direct-правила `geosite:category-ru`;
- наличие direct-правила `geoip:ru`;
- наличие `geoip.dat` и `geosite.dat`;
- размер и дату assets;
- состояние status server;
- health активного proxy path.

Часть логики можно переиспользовать из smoke-тестового контейнера.

## 7. Документация production layout

Описать ожидаемую структуру на сервере:

```text
/opt/proxy-xray/
  docker-compose.yml
  vless-extra.txt
  state.json
  logs/
  backups/
```

Добавить `OPERATIONS.md` с типовыми действиями:

- первая установка;
- обновление;
- rollback;
- просмотр логов;
- запуск smoke-тестов;
- просмотр статуса;
- сброс state;
- добавление локальных VLESS-ссылок.

## 8. История качества серверов

Хранить не только последнее состояние сервера, но и короткую историю качества.

Полезные метрики:

- сколько раз сервер был OK/FAIL за сутки и неделю;
- средняя latency;
- средний throughput;
- когда последний раз была деградация;
- сколько раз сервер был выбран активным;
- сколько успешных переключений было на этот сервер.

Это поможет отличать случайно оживший сервер от действительно стабильного.

## 9. Карантин плохих серверов

Если сервер несколько раз подряд падает, деградирует или дает слишком плохую latency/throughput, временно отправлять его в карантин.

Идея:

- карантин на N часов после серии fail/degraded;
- отдельная причина карантина в `/json` и UI;
- ручная кнопка очистки карантина;
- extra-серверы можно проверять чаще, subscription-серверы беречь.

Плюсы:

- меньше бессмысленных повторных проверок;
- меньше риск переключиться на формально живой, но плохой сервер;
- меньше нагрузка на подписку с ограничением подключений.

## 10. Режимы агрессивности failover

Добавить простую настройку поведения автоматики:

- `conservative`: меньше переключений, терпим краткие просадки;
- `balanced`: текущая логика;
- `aggressive`: быстрее переключаемся при деградации.

Это полезно, потому что для телевизора, браузера и фоновых задач требования разные. Иногда важнее стабильность, иногда скорость реакции.

## 11. Manual pin текущего сервера

Добавить возможность временно закрепить текущий сервер.

Варианты:

- до перезапуска;
- на 1 час;
- пока сервер не сломается;
- до ручной отмены.

Это полезно, когда найден хороший сервер и не хочется, чтобы автоматика слишком активно меняла маршрут.

## 12. Причина выбора активного сервера

На странице статуса рядом с active connection показывать, почему выбран именно этот сервер.

Примеры причин:

- highest score;
- fallback after degradation;
- extra priority;
- last known good;
- selected by Xray balancer;
- subscription servers unavailable;
- pinned manually.

Это упростит разбор ситуаций, когда кажется, что система выбрала не тот сервер.

## 13. Diagnostic bundle одной кнопкой

Добавить сбор диагностического пакета.

Состав:

- текущий `/json`;
- последние логи;
- generated Xray config без секретов;
- состояние `geoip.dat`, `geosite.dat`, `assets-state.json`;
- `docker compose config`;
- live servers;
- all candidates;
- состояние DNS split и RU direct routing.

Это удобно перед переносом на сервер и для случаев, когда "что-то странно тормозит".

## 14. Sanitize секретов

Для экспортов, диагностик и логов добавить маскирование секретов.

Что маскировать:

- Telegram token;
- subscription URL;
- UUID в VLESS-ссылках;
- приватные VLESS links;
- потенциальные query-параметры с ключами.

Цель: можно безопасно отправить diagnostic bundle или кусок логов без риска утечки доступа.

## 15. Легкий audit конфигурации

Добавить endpoint или команду проверки конфигурации.

Проверять:

- subscription URL задан;
- `vless-extra.txt` доступен;
- `state.json` writable;
- assets directory writable;
- DNS split включен;
- RU direct rules есть;
- Telegram chat/token заданы, если включены уведомления;
- exposed ports совпадают с документацией;
- status server доступен.

## 16. Graceful startup profile

Сделать старт контейнера более предсказуемым и быстрым.

Желаемый порядок:

- seed assets из образа;
- загрузить cached candidates из `state.json`;
- запустить Xray с last known good;
- асинхронно обновить подписку;
- асинхронно обновить assets;
- асинхронно запустить candidate checks.

Главная цель: после `docker compose up` интернет должен подняться как можно быстрее из кеша, даже если внешние ресурсы тормозят.

## 17. Минимальный local API для UI actions

Если добавлять кнопки на странице статуса, лучше сразу сделать простой API.

Возможные endpoints:

- `POST /actions/subscription-refresh`;
- `POST /actions/check-random`;
- `POST /actions/restart-xray`;
- `POST /actions/clear-failed`;
- `GET /export/state`;
- `GET /export/diagnostics`.

Так как страница LAN-only, авторизация не нужна, но действия должны быть через `POST`, а UI должен показывать результат операции.

## 18. Эксперименты с TLS fingerprint и ClientHello fragment

Добавить экспериментальные настройки для поведения TLS-handshake против DPI.

Идеи:

- по умолчанию сохранять `fp`, который пришел из подписки;
- разрешить управляемый override fingerprint для extra-серверов и hot-standby slot-ов;
- использовать стабильные профили вроде `chrome`, `firefox`, `safari`, а не менять отпечаток на каждом соединении;
- проверить поведение Xray `fragment` для `tlshello` и понять, применимо ли это чисто к нашим VLESS TLS/REALITY outbound;
- сделать A/B-режим: один hot-standby slot работает с обычным handshake, второй с fragment/fingerprint изменениями.

Ограничения:

- по умолчанию выключено;
- сначала тестировать на extra/local серверах;
- показывать fingerprint и fragment mode в `/json` и UI;
- оценивать не только доступность, но и latency/throughput.

Риск: смена fingerprint и разбиение ClientHello может помочь против некоторых DPI, но может добавить задержку, ломать отдельные маршруты или выглядеть менее естественно. Это нужно внедрять как экспериментальную опцию, а не как default-поведение.
